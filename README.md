## 프로젝트 소개

### 주제

신입사원 한 명이 회사에 입사합니다. 이 사원에게는 **성향**이 부여됩니다.
5가지 성향의 에이전트가 **동일한 회사 환경**에서 20년간(7,300일) 회사 생활을 합니다.

**핵심 질문: "어떤 성향이 회사에서 가장 오래, 가장 높이 올라가는가?"**

### 목적

딥 에이전트 기능 심화 학습 (Reflection)

단순히 LLM에게 질문하고 답을 받는 게 아니라, LLM이 주어진 기간동안 자율적으로 행동하고 그 결과를 관찰하는 구조

### 5가지 성향

| 성향 | 설명 |
|------|------|
| **균형형** | 모든 스탯을 고르게 올리는 타입 |
| **성과형** | 업무 능력과 성과에 올인하는 타입 |
| **사교형** | 동료 관계와 평판에 집중하는 타입 |
| **정치형** | 상사 라인을 타고 정치력으로 승부하는 타입 |
| **워라밸형** | 스트레스 관리와 체력 유지에 집중하는 타입 |

### 실험 구성

- **5가지 성향 비교** — 어떤 성향이 승진하고, 어떤 성향이 해고되는지
- **Reflection(자기성찰) on/off 비교** — 일정 기간마다 자기 상태를 돌아보고 전략을 수정하는 기능이 실제로 효과가 있는지, 같은 성향끼리 비교 실험

## 아키텍처

> 핵심 흐름: 매일 루프 → 30일 배치 결정(LLM) → 90일 Reflection(LLM) → 환경 처리 → 종료 판정

![서비스 흐름](architecture.png)

1. 초기화 단계에서 에이전트 5개를 병렬 생성
2. 매일 반복 루프에 진입 (Day 1 ~ Day 7,300)
3. 하루의 흐름:
   - **주말/평일 구분**
     - 주말: LLM 호출 없이 성향별 가중치로 활동 자동 선택
     - 평일: 30일치 행동 계획이 남았는지 확인
       - 남아있으면 → 하나 꺼내서 실행
       - 소진됐으면 → 두 가지 LLM 호출:
         - 90일 경과 시 **Reflection LLM** (gpt-4.1): 자기성찰. 승진 요건 대비 현재 스탯 갭 분석, 문제점 진단
         - **배치 결정 LLM** (gpt-4.1-mini): 다음 30일치 행동 계획 생성
   - **행동 실행 → 환경 처리** (스탯 변동, 랜덤 이벤트)
   - **중요한 사건은 메모리에 저장**
   - **종료 판정** (번아웃, 만성 스트레스, 권고사직, 해고 등)

## 시뮬레이션 결과

### 5가지 성향 비교 결과

| 순위 | 성향 | 최종 직급 | 연봉 | 생존 기간 | 결말 |
|------|------|----------|------|----------|------|
| 1위 | **성과형** | 임원 | 3억 | 20년 완주 | 현직유지 |
| 2위 | **사교형** | 임원 | 2.8억 | 20년 완주 | 현직유지 |
| 3위 | **균형형** | 이사 | 1.8억 | 20년 완주 | 정년퇴직 |
| 4위 | 워라밸형 | 과장 | 8,500만 | 15년 | 권고사직 |
| 5위 | 정치형 | 사원 | 4,100만 | 5년 | 권고사직 |

**상위권 분석:**
- **성과형** — 업무 능력과 성과에 올인하는 전략이 승진 요건을 가장 빠르게 충족
- **사교형** — 평판이 승진 요건 중 하나를 안정적으로 충족하고, 동료 관계가 좋으면 이벤트에서도 유리한 효과
- **균형형** — 어디서도 크게 밀리지 않지만, 어느 한 스탯이 확 튀지도 않아서 임원까지는 못 감

**하위권 분석:**
- **워라밸형** — 해고 시점에 승진 스탯을 전부 충족하고 있었지만, 승진 확률 판정에서 3번 탈락하며 권고사직 처리
- **정치형** — 상사신뢰와 정치력은 올라가지만 평판이 바닥(0.2)을 쳐서 대리 승진에 필요한 평판 12를 못 채움

### Reflection A/B 비교 결과

> 가장 빨리 해고되는 **정치형**에 대해서 자기성찰을 할 경우 얼마나 변화가 있을지 실험

| | Reflection ON | Reflection OFF |
|---|---|---|
| 최종 직급 | **부장** | 사원 |
| 연봉 | 1.3억 | 4,100만 |
| 생존 기간 | **17년 (6,206일)** | 5년 (1,825일) |
| 결말 | 희망퇴직 (자발적) | 권고사직 (강제) |

**핵심 메커니즘:**
1. 90일마다 자기 상태를 점검
2. Reflection 모델(gpt-4.1)이 현재 스탯과 승진 요건 사이의 갭을 분석하고, "평판이 12 필요한데 지금 3밖에 안 된다, 정치 행동을 줄이고 동료를 도와서 평판을 올려라"라는 식의 처방을 내림
3. 이 처방이 다음 30일 행동 계획을 짜는 LLM(gpt-4.1-mini)의 프롬프트에 들어감
4. 결과적으로 Reflection이 있는 에이전트는 **정치 행동 일변도에서 벗어나서 평판을 보완**할 수 있었고, 대리 승진을 넘긴 이후로는 꾸준히 상승

> Reflection은 에이전트가 자기 약점을 인식하고 행동을 수정할 수 있게 해주는 기능.
> 특히 성향에 의한 편향이 강한 에이전트(정치형처럼)에게 효과가 큼

다만, 구현 과정이 이 설명처럼 단순하지 않았음 (아래 Reflection 구현 심화 참고)

## 코드 구조

```
main.py 설정 → ThreadPoolExecutor 병렬 실행
  → _run_one(): ReActAgent + CompanyEnvironment 생성
    → run_simulation(): Day 1~7300 루프
      → 주말: env.step_weekend() (LLM 호출 없음)
      → 평일: decide_batch() → _build_memory_section() → LLM 호출 → 30일 계획
        → env.step(): 행동 효과 → 자연 변화 → 랜덤 이벤트 → 생존 판정
        → _store_episode_if_important(): 중요 사건만 메모리 저장
```

### 시작점: main.py

```python
EXPERIMENT_SEED    = 42          # 모든 에이전트가 동일한 이벤트 시퀀스를 경험
MAX_DAYS           = 7300        # 20년
DECISION_INTERVAL  = 30          # 30일치 행동을 한 번에 결정
REFLECTION_INTERVAL = 90         # 성찰 주기 — 분기마다

MODEL_DECISION     = "gpt-4.1-mini"    # 배치 결정용 (저렴/빠름)
MODEL_REFLECTION   = "gpt-4.1"         # Reflection용 (고품질)
```

`ThreadPoolExecutor`로 5개 성향을 동시에 병렬 실행. 각 스레드에서 `_run_one()` → `run_simulation()` 호출.

### 핵심 객체 3가지

| 객체 | 역할 |
|------|------|
| `LLMClient` | OpenAI API 호출 클라이언트. 배치 결정용(gpt-4.1-mini)과 Reflection용(gpt-4.1) |
| `ReActAgent` | 성향이 주입된 에이전트. 시스템 프롬프트에 성향 설명 포함 |
| `CompanyEnvironment` | 회사 환경. 승진 요건, 해고 조건, 랜덤 이벤트 확률 테이블 |

    # decide_batch()를 오버라이드하여 30일치 배치 결정
```

### 3계층 메모리 (`_build_memory_section`)

프롬프트에 들어가는 메모리가 3계층으로 구성:

| 계층 | 내용 | 역할 |
|------|------|------|
| 1층 | Reflection 결과 | `[!][!][!]` 최우선 전략 지침 |
| 2층 | 에피소딕 메모리 | 승진, 해고 위기 등 중요 사건 (최근 10개) |
| 3층 | 히스토리 압축 | 최근 30일 행동 패턴 요약 (LLM 3문장 요약) |

### 환경 처리 (`env.step`)

하루 처리 순서:
1. 행동 효과 적용 (성향 배율 곱셈)
2. 자연 회복/저하 (체력+5, 스트레스-2 등)
3. 랜덤 이벤트 확률 판정 (회사 0.2%/일, 팀 1.2%/일, 개인 4%/일)
4. 모든 스탯 0~100 범위 제한
5. 생존 판정 (번아웃, 해고, 승진, 이직)

### 중요도 판별 (`_classify_outcome`)

7,300일을 전부 저장하면 프롬프트가 터지므로, 아래 6가지에 해당하는 날만 에피소딕 메모리에 저장:

- 승진 / 이직 / 해고 / 랜덤 이벤트 발생 / 번아웃 위기 / 해고 위기

오래된 건 FIFO로 밀려나고 메모리 50개만 유지.

## Reflection 구현 심화

이 기능이 지금 형태가 되기까지 **4번의 구현-실험 사이클**을 거침.

### 구현과정 요약

| 단계 | 구현 | 실험 결과 |
|------|------|-----------|
| **1** | 기본 구조 (단순 프롬프트 + 30일 주기) | Reflect가 오히려 역효과 (5년 vs 6년) |
| **2** | 수치 제공 + 주기 90일 + 처방 형식 | 여전히 동일 (둘 다 5년 해고) |
| **3** | 프롬프트 3곳 강화 (`[!][!][!]`, 강제 원칙) | 야근 남발 → 번아웃 악순환 |
| **4** | 체력/스트레스 관리 원칙 + 희망퇴직 경고 | **17년 부장 vs 5년 사원** |

### 1단계: 기본 구조 구현

Reflection의 뼈대를 구축. 모델을 두 개로 나누어 성찰은 gpt-4.1(고품질), 행동 결정은 gpt-4.1-mini(저렴)가 담당.

```
매 30일 배치 결정 전 → reflect() 호출 (gpt-4.1)
→ 성찰 결과를 _reflection에 저장
→ decide_batch() 호출 (gpt-4.1-mini) → 프롬프트에 [ 자기성찰 결과 ] 포함
```

```
실험 결과:
[1위] 정치형_NoReflect | 사원 | 생존 2,279일 (6년) | 해고
[2위] 정치형_Reflect   | 사원 | 생존 1,958일 (5년) | 해고
```

**Reflection이 있는 쪽이 오히려 더 빨리 해고됨.**

원인:
1. **성찰 내용이 추상적** — 매번 "성과를 올려라" 일반론만 반복. 구체적으로 어떤 스탯이 얼마나 부족한지 모름
2. **전략을 너무 자주 흔듦** — 30일마다 방향을 계속 바꿈. NoReflect는 성향대로 일관되게 행동해서 오히려 안정적
3. **근본 문제를 못 건드림** — 정치형 해고의 핵심 원인은 평판이 0이라 대리 승진을 못 하는 것. 추상적 조언으로 해결 불가

### 2단계: 수치 + 주기 + 형식 개선

세 가지를 동시에 수정:

- **승진 요건 갭 수치를 프롬프트에 제공** (`_build_promotion_gap()`)
  - "평판이 부족하다"가 아니라 **"평판 5인데 20 필요하다, 15 부족하다"**를 알려줌
- **성찰 주기 30일 → 90일**
- **출력 형식을 처방 형태로 구체화** (행동별 일수 배분)

```
실험 결과:
[1위] 정치형_NoReflect | 사원 | 생존 1,825일 (5년) | 해고
[2위] 정치형_Reflect   | 사원 | 생존 1,825일 (5년) | 해고
```

**개선했는데 결과가 똑같음.** 구조적 문제:
1. **Reflection 결과가 너무 약하게 전달됨** — 같은 프롬프트에 성향 설명도 있어서, gpt-4.1-mini가 성향 설명을 더 우선시
2. **Reflection 내용이 행동으로 강제되지 않음** — 그냥 참고용 텍스트일 뿐, 실제 행동을 바꾸는 메커니즘이 없음

→ Reflection LLM(gpt-4.1)이 아무리 좋은 분석을 해도, 배치 결정 LLM(gpt-4.1-mini)이 그걸 무시하면 의미 없음

### 3단계: 프롬프트 3곳 동시 강화

근본적인 프롬프트 개편:

1. **Reflection 프롬프트 전면 재설계** — 역할을 "자기성찰 모듈" → "전략 컨설턴트"로 격상, 행동별 효과 참고표 추가
2. **memory_section에서 Reflection 격상** — `[!][!][!]` 태그 + "반드시 따르세요" + "무시하면 해고됩니다"
3. **배치 결정 프롬프트에 강제 원칙 추가** — "최우선 전략 지침이 있다면 반드시 따르라"는 명시적 지시

```
실험 결과:
정치형_Reflect   | 대리 | 생존 1,112일 (3년) | 만성 스트레스 퇴사
정치형_NoReflect | 대리 | 생존 1,825일 (5년) | 정년퇴직
```

mini 모델이 처방을 따르기 시작했지만, **새로운 문제 발생:**

Reflection이 "성과 부족! 야근해!" 처방 → LLM이 처방을 따름 → 고강도 행동 연속 실행 → **스트레스 폭증, 체력 바닥 → 만성 스트레스 퇴사**

처방이 **승진 요건만 보고 스트레스/체력 관리를 무시**한 것.

### 4단계: 체력/스트레스 관리 원칙 추가 — 최종 성공

Reflection 프롬프트에 **생존 관련 안전장치** 추가:

1. **희망퇴직 확률 경고** — 스트레스/체력 상태에 따른 퇴직 확률을 구체적 수치로 제공
2. **체력/스트레스 관리 필수 원칙** — "장기 생존이 승진보다 중요합니다. 죽으면 승진도 없습니다."
3. **행동 효과 강조도 변경** — 야근: `남용 금지!`, 휴가: `생존 핵심!`

```
최종 실험 결과:

Reflection이 "평판 0.2인데 12 필요하다, 동료를 도와줘서 올려라"고 처방하고, 배치 결정 LLM이 그걸 실제로 따름.
정치 일변도에서 벗어나 평판을 보완하면서도, 체력/스트레스 관리 원칙 덕분에 번아웃 없이 안정적으로 성장.

### 핵심 교훈

1. **Reflection에 구체적인 수치를 제공해야 한다** (1→2단계)
   - "잘해라"가 아니라 "평판 5인데 20 필요, 15 부족"을 알려줘야 함

2. **Reflection 결과를 텍스트로만 전달하면 의사결정 모델이 무시할 수 있다** (2→3단계)
   - 특히 저비용 모델(mini)일수록 긴 시스템 프롬프트의 지시를 놓침 (강제 지시 태그와 배치 프롬프트 보강 필요)

3. **승진만 보면 안 된다 — 생존 원칙이 필요하다** (3→4단계)
   - "성과 올려라 → 야근 → 번아웃"의 악순환. Reflection이 승진과 생존을 동시에 고려하게 만들어야 함

4. **그래도 근본적 한계는 있다**
   - 이 모든 수정은 전부 **프롬프트 텍스트 강화**. gpt-4.1-mini가 지시를 따를지는 여전히 모델 판단에 의존
   - 진짜 확실하게 하려면 **구조적으로 행동에 영향을 미치는 설계** 필요 (예: Reflection 결과로 행동 확률 가중치를 직접 수정하거나, 특정 행동을 강제로 n일 예약하는 방식)

> **LLM에게 "하라고 텍스트로 말하는 것"과 "구조적으로 하게 만드는 것"은 다르다.**
